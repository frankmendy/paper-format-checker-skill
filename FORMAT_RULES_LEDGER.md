# 论文格式自动修正检查项记录 (Format Rules Ledger)

本文档用于记录在自动化论文格式修正过程中，通过人工沟通发现并修复的具体格式问题。这些规则已硬编码到修正脚本中，作为后续所有论文格式修正的**强制检查项**。

## 1. 摘要与关键词部分 (Abstract & Keywords)

### 1.1 标题与内容的分离与加粗处理
- **问题描述：** Word 中“摘要：”和摘要内容通常在同一个段落 (Paragraph) 中，如果统一应用段落样式，会导致内容被意外加粗。
- **强制规则：** 
  - **中文摘要/关键词：** 仅标题（如“摘 要”、“关键词”）使用 **五号黑体，加粗**。一旦遇到冒号（`:` 或 `：`），后续内容必须强制取消加粗，并使用 **五号楷体**。
  - **英文 Abstract/Key words：** 仅标题使用 **小四号 Times New Roman，加粗**。冒号之后的内容必须强制取消加粗，并使用 **小四号 Times New Roman**。

### 1.2 摘要部分的对齐方式
- **问题描述：** 模板要求摘要段落不能仅仅是左对齐。
- **强制规则：** 中英文摘要及关键词的整个段落必须使用 **两端对齐 (Justify)**。

### 1.4 摘要内容范围判定
- **问题描述：** 部分论文的摘要正文并不和“摘要：”“ABSTRACT:”写在同一段，而是从下一段开始，直到“关键词/KEYWORDS”之前结束。若只修标题行，会导致中间的摘要正文仍保留正文样式。
- **强制规则：**
  - **中文摘要内容范围**：从“摘要”标题所在段之后开始，到“关键词”标题所在段之前结束；该范围内所有非标题段落都必须按**中文摘要内容格式**处理。
  - **英文摘要内容范围**：从“ABSTRACT”标题所在段之后开始，到“KEYWORDS”标题所在段之前结束；该范围内所有非标题段落都必须按**英文摘要内容格式**处理。
  - **实现要求**：不能只依赖段首是否写着“摘要/ABSTRACT”来判定；必须结合段落索引范围做连续处理与复检。

### 1.3 摘要标题的空格容错
- **问题描述：** 中文“摘要”二字中间可能包含空格（如“摘 要”或“摘　要”），导致精确匹配失败。
- **强制规则：** 脚本在匹配特征词之前，必须先清除段落文本中的所有半角和全角空格。

## 2. 目录部分 (Table of Contents)

### 2.1 目录格式规则
**定位方式**：从文档开始，直到遇到包含“第一章”且样式为“Heading 1”的正文段落之前的区域。
**规则细节**：
- **字体与字号**：所有层级的目录项均使用“宋体”，12磅（小四号）。
- **目录标题**：目录标题必须统一规范为 **“目    录”**（“目”和“录”之间四个半角空格），并且**绝对居中**。
- **颜色与装饰**：所有目录项字体颜色强制为黑色（`RGB(0,0,0)`），且**绝对不允许**有下划线（包括超链接带来的默认蓝色下划线）、倾斜或高亮。
- **行间距**：**1.5 倍行距**，段前段后 0 磅。
- **对齐与页码**：所有目录项左对齐。页码必须位于行末，右对齐，且由前导点（`..........`）连接。
- **层级缩进（绝对值）**：
  - **一级标题**（如：第一章、参考文献、致谢、附录等）：左侧缩进 `0 cm`，**加粗**。编号和文字之间强制保留1个半角空格。特别注意：“致谢”二字中间**必须强制包含四个半角空格**（即格式化为“致    谢”）。
  - **二级标题**（如：1.1）：左侧缩进 `0.37 cm`（约1字符），不加粗。编号和文字之间**无空格**（紧贴）。
  - **三级标题**（如：1.1.1）：左侧缩进 `0.74 cm`（约2字符），不加粗。编号和文字之间**无空格**（紧贴）。
  - **编号点号兼容**：编号中的点号可能是半角 `.` 或全角 `．`，脚本必须统一归一化为半角 `.` 后再做层级判断与缩进设置，否则二/三级标题会被误判层级导致格式错乱。

**排坑经验（目录没有“目录”标题但存在 TOC 条目）**：
- **问题**：部分文档导出的目录不包含单独的“目录/目 录”标题，但目录条目仍然存在（常见为 Word 的 `TOC 1/2/3` 段落样式，或“标题 + 制表符 + 页码数字”的形式）。如果只用“找到标题=目录”作为 TOC 起点，会导致目录条目被误当作正文处理，从而把目录格式整体改乱。
- **解决方案**：
  1. TOC 起点判定需要兜底：只要遇到 `TOC 1/2/3`（大小写与空格不敏感）样式段落，也认为进入目录区域。
  2. 对于已被脚本重建为纯文本的目录项（无 `TOC` 样式），继续用“包含制表符且末尾为页码数字”的模式将其纳入目录范围，确保循环复检多轮后目录范围稳定。
  3. 若目录区域内缺少“目  录”标题，则自动插入一行“目  录”，并按模板（三号宋体加粗、居中）格式化。

**排坑经验（Word 自动目录存在，但 python-docx 看不到）**：
- **问题**：部分论文的目录是 Word 自动生成的域目录，底层位于域代码/超链接/内容控件结构中。Word 中能看到完整目录，但 `python-docx` 读取普通段落时，可能既读不到“目录”标题，也读不到 `TOC 1/2/3` 样式段落，因此阶段 2 会误判为“没有目录”。
- **识别信号**：
  1. `document.xml` 中存在 `TOC \o`、`PAGEREF`、`_Tocxxxx` 等字段标记；
  2. Word 自动化接口 `doc.TablesOfContents.Count > 0`。
- **解决方案**：
  1. 目录识别不能只依赖 `python-docx` 的 `doc.paragraphs`。
  2. 在 Windows + Word 环境下，必须增加 `doc.TablesOfContents` 级别的兜底识别。
  3. 对识别到的 Word 自动目录，优先在 Word 自动化阶段直接按 TOC Range 逐段修正字体、缩进、加粗和行距，再执行 `toc.Update()` 更新页码。
  4. 为了避免日志中出现“目录计数 0”的误导，修复阶段还应额外从 `document.xml` 中统计 `PAGEREF _Toc` 数量，把它作为 Word 自动目录项数量输出到阶段 2 统计中。

**排坑经验（Word 自动目录标题可见，但常规段落接口改不动）**：
- **问题**：目录标题“目    录”在 Word 中可见，但通过普通段落接口修改后，仍可能保持左对齐、12 磅、非加粗。
- **原因**：该标题段有时与 Word 自动目录域处于同一内容控件/特殊结构附近，`python-docx` 看不到，Word COM 的普通 Range 格式化也可能不稳定落盘。
- **解决方案**：在 Word 更新完成并关闭文档后，直接对 `document.xml` 做一次兜底后处理：定位 `TOC \o` 字段前一段，将其强制重写为 **“目    录” + 居中 + 宋体三号加粗**。

**排坑经验（目录层级看起来不对，根源其实在正文标题级别）**：
- **排坑经验（目录层级看起来不对，根源其实在正文标题级别）**：
- **问题**：即使识别到了 Word 自动目录，目录中的 `1.2.1 / 1.2.2 / 1.2.3` 等条目仍可能显示成一级目录格式（加粗、无缩进），看起来像“目录没按模板修好”。
- **原因**：Word 自动目录的层级首先继承的是**正文源标题的级别**。如果正文里的 `1.2.1` 被错误套成 `Heading 1` 或被脚本误判为二级标题，那么更新目录后，它生成的仍会是一级 TOC 条目。此外，如果正文中的二/三级标题被设置了**加粗**（如小三加粗/四号加粗），这些加粗属性会被 Word 原封不动地带入目录，导致目录中的二/三级条目也变成加粗，违反目录层级规范。
- **解决方案**：
  1. 正文标题识别时必须先判断三级标题，再判断二级标题，避免 `1.2.1` 被 `1.2` 的规则提前匹配。
  2. 对正文标题不仅要修字体/对齐，还要同步修正其 `Heading 1 / Heading 2 / Heading 3` 样式级别。
  3. **强制取消正文中二/三级标题的加粗属性**（或仅在生成目录前取消，但最佳实践是正文和目录的二/三级均不加粗，以保证更新生成的 TOC 2 / TOC 3 是正常粗细）。
  4. 更新目录前先修正正文标题级别，目录才会在 Word 中生成正确的 TOC 1 / TOC 2 / TOC 3 层级。

**排坑经验（一级目录标题未加粗/被覆盖）**：
- **问题**：在修复过程中，目录的一级标题（如“第一章”）有时会丢失加粗属性。
- **原因**：在处理目录超链接以消除蓝色下划线时，脚本遍历了段落中的所有 `Hyperlink` 对象并重置了它们的字体属性（包括 `Bold = False`）。这直接覆盖了我们刚刚给一级目录段落施加的加粗效果。
- **解决方案**：在清除超链接下划线的循环内部，增加一次层级判断。如果当前处理的超链接文本属于一级目录（如匹配“第X章”或“参考文献”等），则必须再次强行设置 `h_range.Font.Bold = True`。

### 2.3 目录页码一致性（Word 域更新）
- **问题描述**：脚本在 python-docx 层只能修改文本/样式，无法像 Word 排版引擎那样计算真实分页，因此目录中的页码可能与修复后的真实页码不一致。
- **强制规则**：修复完成保存后，在 Windows 环境下应尝试调用本机 Word 自动更新域（`Fields.Update()`/`TablesOfContents.Update()`）并保存，确保目录页码与正文真实页码一致。若环境缺少 Word 或缺少 `pywin32`，则降级为提示用户手动在 Word 中更新目录。

### 2.4 页码分节与编号格式
- **目录独立成节**：目录必须独立成页（分节，Next Page section break）。目录页码使用**罗马数字**，并在目录每一页页脚**居中显示**。
- **正文页码重置**：正文必须使用**阿拉伯数字**，并在正文每一页页脚**居中显示**；正文所在分节必须将页码**从 1 开始**重新编号（RestartNumberingAtSection + StartingNumber=1）。
- **页码字体**：页脚页码必须统一为 **Times New Roman**。
- **避免空白页/空行**：插入分节符时必须在“上一段落末尾”插入（CollapseEnd + InsertBreak），并在正文开始段落前清理可能残留的空段落，保证目录结束后下一页直接顶格进入“第一章”。
- **避免页码重复**：必须清空页脚原有内容/字段（不仅仅删除 PageNumbers 集合），再重新添加页码字段，防止出现两个页码同时存在的重复显示。
- **实现方式**：Windows 环境下优先使用本机 Word 自动化插入分节符并设置页码格式（page number style + restart），然后再更新目录域与分页。
- **终极重建排坑指南**（处理 Word 自动生成目录的深坑）：
  1. **间距幽灵**：目录中编号和文字之间的距离往往不是简单的空格字符，而是被隐藏的制表符（`\t`）、不间断空格（`\xA0`），甚至是段落的**多级列表（`<w:numPr>`）和悬挂缩进（`hanging`）**造成的。必须在底层 XML 彻底抹除这些属性。
  2. **下划线幽灵**：特殊的无编号标题（如参考文献）常带有顽固的超链接下划线。它可能隐藏在字符属性（`<w:rPr>`）中，也可能污染了整个段落标记的属性（`pPr` 中的 `rPr`），甚至是全局 `Normal` 样式。
  3. **制表位对齐幽灵**：硬编码的制表位（如 `8200`）无法完美适应不同文档的页边距，导致页码无法与正文右边距绝对对齐。必须动态读取当前节的页面宽度和左右边距，计算出精确的可用宽度（如 A4 标准为 `8312 twips`），并将制表位的 `pos` 属性精准设置为该值。
  4. **唯一解法（破而后立）**：不要在原段落上修补。必须提取纯文本 -> 删除原段落 -> 新建纯净段落 -> 解除任何样式绑定（`style = None`） -> 物理删除段落级和字符级的 `<w:numPr>`、`<w:rPr>`、`<w:u>` 节点 -> 重新赋值阶梯缩进、动态制表位和纯文本。

**排坑经验（正文小标题包含“目录”导致误判为 TOC）**：
- **问题**：某些正文段落（如“2 目录与文件结构”）由于包含“目录”二字，且长度较短，会被误判为进入目录部分，从而导致后续正文段落被当作目录检查，出现“目录第 X 段未左对齐”等假阳性。
- **原因**：目录进入条件若使用“包含 `目录` 且长度 < 10”这类模糊匹配，会在没有真实 TOC 的文档中误触发。
- **解决方案**：
  1. 进入目录部分必须使用严格条件：清理空格后段落文本 **等于** `目录`（允许“目  录”这种空格变体经清理后归一化为 `目录`），而不是“包含目录”。
  2. 相同的分段逻辑需要同时修正“修复阶段”和“复检阶段”，避免修复与校验状态机不一致。

### 2.2 附录 (Appendix) 的动态联动
- **问题描述：** 有些论文的目录中写了“附录”，但正文末尾并没有附录内容。
- **强制规则：** 修正脚本在处理目录前必须全局扫描一次正文。如果正文中不存在“附录”章节，则**自动删除**目录中的“附录”条目。

## 3. 标题识别与定位 (Titles Recognition)

### 3.1 正文标题 (Titles) 格式规范
**规则细节**：
- **正文一级标题**（如“第一章 绪论”）：必须**绝对居中**。三号宋体，加粗，段前段后各1行（16磅）。
- **正文二级标题**（如“1.1”）：小三号宋体，加粗，左对齐，**无缩进**。

### 3.2 正文段落 (Body Text) 格式规范
- **对齐方式**：正文普通段落必须使用 **两端对齐 (Justify)**（不要用左对齐替代）。
- **首行缩进**：正文普通段落首行缩进 2 字符（脚本按 `Pt(24)` 近似实现）。
- **行距**：固定值 20 磅。
- **英文字体分离控制**：即使正文整体要求为“宋体”，对于正文中的英文/数字内容，必须强制使用 `Times New Roman`（底层 `rFonts` 中配置 `w:ascii` 和 `w:hAnsi` 为 `Times New Roman`），而中文部分则配置 `w:eastAsia` 为宋体。

**排坑经验（一级标题无法居中/偏右的根源）**：
- **问题**：在给一级标题应用居中对齐（`WD_ALIGN_PARAGRAPH.CENTER`）后，发现标题视觉上依然偏右。
- **原因**：由于一级标题往往是由正文修改而来，或者继承了 `Normal` / `Body Text` 样式，它底层通常带有了“首行缩进 2 字符”或“左侧缩进”的属性。在 Word 的排版逻辑中，居中是对“可用宽度（扣除缩进后）”进行的居中，所以一旦有缩进，绝对中心就会发生偏移。
- **解决方案**：在设置对齐之前，必须执行双重清理：
  1. 调用底层 XML 清理函数 `_clear_paragraph_indents(para)` 抹除所有的 `<w:ind>` 属性（首行、悬挂）。
  2. 显式在 Python API 层面设置 `para.paragraph_format.left_indent = Cm(0)` 和 `first_line_indent = Cm(0)`，确保其彻底归零，从而实现绝对居中。

**排坑经验（一级标题被当成正文段落，出现“两端对齐 + 首行缩进 2 字符”）**：
- **问题**：文档中“第一章/第二章/第三章…”等一级标题没有被修复为居中加粗三号宋体，反而表现为正文样式（两端对齐、首行缩进 2 字符）。
- **原因**：
  1. 文档的一级标题可能不是严格的 `第X章` 纯文本格式（例如存在额外空格：`第  三  章`，或标题不是 `第X章` 而是 `Heading 1` 样式的“引言”）。
  2. 若一级标题识别仅依赖 `^第X章`，会漏掉 `Heading 1` 样式的正文章节标题，导致其落入“普通正文”分支，被强制两端对齐和首行缩进。
- **解决方案**：
  1. 一级标题识别必须同时支持两种兜底：`Heading 1` 样式 或 `第\s*X\s*章`（允许任意空格）。
  2. 一旦识别为一级标题，必须清零缩进并强制 `w:jc="center"`，避免被正文对齐逻辑覆盖。
  3. 对 `第X章` 标题做规范化：统一输出为 `第X章 标题`（“章”后保留一个半角空格）。

**排坑经验（二、三级标题漏识别与格式未套用）**：
- **问题**：部分论文中的二、三级标题（如 `1.1研究背景`）既没有被加粗，也没有取消首行缩进，完全被当成了普通正文处理。
- **原因**：之前的识别正则使用了单词边界符 `\b`（如 `^\d+\.\d+\b`），这在标题编号和中文文字之间有空格时（`1.1 研究`）可以匹配；但如果编号和中文紧贴（`1.1研究`），数字后紧跟中文字符就不构成单词边界，导致正则匹配失败，标题被漏修。
- **解决方案**：
  1. 移除正则表达式中的 `\b`，改为使用前向否定断言（负向先行断言）或直接放宽匹配条件。
  2. 二级标题正则改为：`r'^\d+[\.．]\d+(?![\.．]\d)'`
  3. 三级标题正则改为：`r'^\d+[\.．]\d+[\.．]\d+'`
  4. 这样无论是 `1.1研究` 还是 `1.1 研究` 还是 `1.1\t研究`，都能被稳定识别。

**排坑经验（二、三级标题看似左对齐却仍残留首行缩进）**：
- **问题**：二级标题和三级标题虽然被设置成了左对齐，但视觉上依然向右缩进了一截。
- **原因**：仅设置 `alignment=LEFT` 或 `first_line_indent=0` 不足以消除样式继承。二、三级标题同样可能继承正文段落或标题样式里的 `<w:ind>` 节点，导致底层仍保留首行缩进、悬挂缩进或左侧缩进。
- **解决方案**：
  1. 在处理二、三级标题前，先调用 `_clear_paragraph_indents(para)` 清空 XML 层面的全部缩进。
  2. 再显式设置 `first_line_indent = Cm(0)`、`left_indent = Cm(0)`。
  3. 最后额外在段落属性中写入 `w:jc="left"`，确保 Word 按绝对左对齐渲染。

**特殊一级标题与参考文献规则**：
- **参考文献、致谢标题**：正文中的“参考文献”和“致谢”必须按**一级标题**处理，即**三号宋体、加粗、居中、单倍行距、段前段后各 1 行**。  
- **正文一级标题唯一标识**：正文一级标题必须以“**第X章**”（X 可为中文数字或阿拉伯数字）作为唯一标识，不以标题内容（如“引言”等）作为识别条件。
- **参考文献正文**：每一条参考文献必须为**五号楷体、两端对齐、顶格、固定行距 20 磅、不得加粗**，不能只检查“楷体”这一项。

**排坑经验（参考文献看起来不对，但校验没报错）**：
- **问题**：参考文献段落视觉上与模板不一致，但旧版校验没有报错。
- **原因**：旧校验只检查了“是否是楷体”，没有继续检查字号、是否加粗、是否两端对齐、是否顶格、是否为固定 20 磅行距，因此会漏报。
- **解决方案**：参考文献校验与修复必须同时覆盖 **字体、字号、加粗、对齐、缩进、行距** 六个维度；Word 更新阶段也要再对“参考文献”章节正文做一次强制落地。

**排坑经验（目录结束与正文开始的兜底判定）**：
- **问题**：部分文档的正文虽然包含目录，但正文开始处可能不是“第X章”开头（例如用 `Heading 1` 样式但标题文本为其他形式），若仅以“第X章”作为 TOC→正文切换条件，会导致目录段落范围过大，正文被误当目录处理。
- **解决方案**：TOC→正文切换允许使用“样式兜底”：在目录区域内遇到 `Heading 1` 且末尾不是页码数字的段落时，认为已进入正文区域（仅用于分段状态机切换，不用于一级标题格式化）。
- **致谢标题文本**：必须强制规范化为 **“致    谢”**，即“致”和“谢”之间保留**四个半角空格**。
- **参考文献内容**：必须按模板统一为**五号楷体、顶格、两端对齐、固定行距 20 磅**，不得继承正文首行缩进。

**排坑经验（参考文献和致谢看似识别到了，却没有完全套用模板）**：
- **问题**：正文中的“参考文献”“致谢”虽然被识别成特殊段落，但标题格式仍可能沿用 `Title` 或旧样式；参考文献正文也可能只改了字体，未清理缩进和行距。
- **原因**：仅靠文本匹配不足以完全覆盖模板要求，特别是原文档里的“参考文献”“致谢”常带有旧样式残留；另外参考文献正文不一定全部以 `[1]` 开头，若不按“当前正文上下文”持续识别，就会漏修。
- **解决方案**：
  1. 对“参考文献”“致谢”标题统一走一级标题修复逻辑，并清空 XML 缩进、强制写入居中对齐。
  2. 对“致谢”标题文本进行标准化，统一改写成“致    谢”。
  3. 在正文遍历阶段增加 `_body_context`，进入“参考文献”标题后，后续段落持续按参考文献正文格式处理，直到遇到下一个一级标题或特殊章节标题。
  4. 对参考文献正文同时强制应用**楷体五号 + 固定 20 磅行距 + 顶格**，而不是只改其中一项。

### 3.2 图表与题注格式
**规则细节**：
- **图片与表格段落**：只要段落中包含图片（`<w:drawing>` 或 `<v:imagedata>`）或表格，必须**绝对居中**（`WD_ALIGN_PARAGRAPH.CENTER`），且**强制去除任何首行缩进或左侧缩进**。
- **图名格式**：图名必须位于图片**下方**，使用**五号楷体**，**1.5 倍行距**，居中，且**不允许首行缩进**。
- **表名格式**：表名必须位于表格**上方**，使用**五号楷体**，**1.5 倍行距**，居中，且**不允许首行缩进**。
- **表格内容格式**：表格单元格内容统一使用**五号宋体**，**1.5 倍行距**，单元格内**水平居中 + 垂直居中**，且段落**不允许首行缩进或左缩进**。
 - **表格整体居中**：表格对象本身必须在页面中居中（`table.alignment = CENTER`）。仅设置单元格内文字居中并不能保证表格整体居中。

**排坑经验（图片明明处理了却仍未居中的根源）**：
- **问题**：部分图片上方的图像仍然没有居中，但对应的图注已经是居中的。
- **原因**：这些图片所在的段落往往是**纯空段落**，`para.text.strip()` 结果为空。旧逻辑在正文遍历阶段对空段落直接 `continue`，导致这些真正承载图片的段落压根没有进入 `_fix_body_para()`，自然也就没有执行居中修复。
- **进一步特征**：这类图片常见于图 2.2-1、图 2.2-2、图 2.2-4、图 2.2-5 这类图注上方的空白图片段落，底层 XML 虽然包含 `<w:drawing>`、`<w:pict>`、`<v:imagedata>` 等图片节点，但文本层面是空的。
- **解决方案**：
  1. 在正文主循环中，不能只根据 `text` 是否为空来跳过段落。
  2. 必须先检查该空段落的完整 XML 中是否存在图片相关标签（如 `<w:drawing>`、`<v:imagedata>`、`<pic>`、`<w:object>`、`<w:pict>`）。
  3. 只要命中这些标签，即使 `text` 为空，也必须继续执行 `_fix_body_para()`。
  4. 在 `_fix_body_para()` 中，再同时执行缩进清理、段落级 `jc=center` 注入和 Python API 对齐设置，才能确保图片绝对居中。

**排坑经验（图例和表格名称已经居中但仍残留首行缩进）**：
- **问题**：图注、表名虽然设置成了居中，但整行仍然带着正文的首行缩进。
- **原因**：题注往往在后续 `_fix_captions()` 阶段再次套用了 `Normal` 样式，若不额外清理底层缩进，就会把正文首行缩进重新带回来。
- **解决方案**：
  1. 在 `_fix_body_para()` 和 `_fix_captions()` 两个阶段都执行 `_clear_paragraph_indents(para)`。
  2. 显式设置 `first_line_indent = Cm(0)`、`left_indent = Cm(0)`。
  3. 最后强制写入 `w:jc="center"`，避免只在视觉层设置居中而底层仍保留缩进。

**排坑经验（图名规则之前改过，但修复时仍漏掉）**：
- **问题**：图名本应是**五号楷体、位于图下、1.5 倍行距、居中**，但旧版修复后仍可能保留成正文字体，复检也没有报错。
- **原因**：
  1. 旧台账对图题注写法不够明确，曾保留“宋体（或楷体，视具体要求）”这种模糊描述；
  2. 旧代码只对**表名**强制改成楷体五号，没有对**图名**做同样处理；
  3. 旧复检只检查图名是否居中、是否清零缩进，没有检查**字体、字号、行距、是否位于图下**。
- **解决方案**：
  1. 将规则明确收敛为：**图名统一五号楷体，且必须位于图片下方**。
  2. 在 `_fix_captions()` 中对图名与表名统一施加**楷体五号 + 1.5 倍行距 + 居中 + 零缩进**。
  3. 若发现图名在图片上方（下一个相邻段落才是图片段落），则自动把图名移动到图片下方。
  4. 复检时新增图名字体、字号、行距及“位于图下”的检查，避免再次漏报。

**排坑经验（表名出现在表格下方但旧逻辑没报错）**：
- **问题**：像“表6-6 检测耗时统计表”这类表名实际位于表格下方，但旧逻辑只会向上查找表名前一个段落，因此既不会把它移到表格上方，也不会在复检中报错。
- **原因**：`_find_table_caption_paragraph()` 只检查表格前面的段落，没有检查表格后紧邻的段落是否是表名。
- **解决方案**：
  1. 表格修复时必须同时检查表格**上方**和**下方**的相邻段落。
  2. 若下方段落是合法表名（如 `表6-6 ...`），必须将该段落整体移动到表格上方，再应用表名格式。
  3. 复检时若发现表格下方紧邻段落仍是表名，必须直接报错，不能再视为通过。

**排坑经验（表名和表格内容看起来“像表格”，但样式仍与模板不一致）**：
- **问题**：表名字体仍沿用普通正文或 `Normal` 样式；表格单元格内容的字体、字号、行距和对齐方式也未统一。
- **原因**：Word 表格的样式并不会自动随着正文段落格式修复而同步。表名是独立段落，表格内容则藏在 `cell.paragraphs` 里；如果只修正文段落，不单独遍历 `doc.tables`，这些内容就不会被触达。
- **解决方案**：
  1. 单独遍历 `doc.tables`，逐个处理每个单元格内的段落和 run。
  2. 对表名：定位表格前一个段落，强制改为**楷体五号、1.5 倍行距、居中、零缩进**。
  3. 对表格内容：统一改为**宋体五号、1.5 倍行距、水平居中、垂直居中、零缩进**。
  4. 对中文 run 不能只设 `run.font.name`，还要同步写入 `w:eastAsia`，否则 Word 可能仍沿用旧的中文字体。

### 3.2 英文标题 (English Title) 的精准识别
- **问题描述：** 英文标题通常没有明显的特征词，容易被漏判或与普通正文混淆。
- **强制规则：** 利用上下文定位机制。扫描全文，若发现位于英文 "Abstract" 段落**上方**的非空纯英文段落，则将其识别为英文标题。
- **格式要求：** 强制应用 **三号 Times New Roman，加粗，居中**。

### 2.2 标题与正文的兜底区分
- **问题描述：** 部分包含“摘要”等关键字的正文段落可能被错误识别为论文题目。
- **强制规则：** 题目的识别必须放置在特征匹配的最后，并且应增加长度限制（如字数小于50字）来避免误判正文。

## 3. 段落间距与空行 (Spacing & Empty Lines)

### 3.1 模板特定的空行高度控制
- **问题描述：** 模板中明确规定了题目与摘要之间等位置的空行需要有特定的高度（如“五号字空一行”）。
- **强制规则：** 脚本遇到空行段落时不能直接跳过，必须强制将其字号设置为 **10.5 pt（五号字）**，以确保空行高度完全符合模板规范。
